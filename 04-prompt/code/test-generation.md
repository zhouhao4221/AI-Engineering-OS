# 测试编写提示

## 目的

为已有或新生成的代码编写测试用例。适用于工具函数、API 适配器、服务类的测试初稿。核心业务逻辑的测试须人工编写，本提示生成的测试需要人工审查边界情况。

## 输入

- `{{被测代码}}`：要编写测试的函数或模块
- `{{接口定义}}`：类型、函数签名、枚举（若与被测代码分离）
- `{{测试框架}}`：使用的测试框架（如 Jest、pytest、Go test）
- `{{边界条件}}`：需要重点覆盖的边界情况（可留空，AI 自行识别）

## 提示

```
你是一名注重边界情况的测试工程师。请为以下代码编写测试用例。

被测代码：
{{被测代码}}

接口定义（若有）：
{{接口定义}}

测试框架：{{测试框架}}

需覆盖的边界条件（若留空则自行识别）：
{{边界条件}}

要求：
- 覆盖 happy path（正常输入 → 正常输出）
- 覆盖边界值：空值、零值、最大值、类型边界
- 覆盖错误路径：非法输入、依赖失败、权限不足等
- 每个测试用例只测一件事，用描述性名称说明场景
- 不要 mock 核心业务逻辑，只 mock 外部依赖（网络、数据库、时间）
- 不要生成永远通过的断言（如 assertTrue(true)）

先列出你识别的测试场景清单，再输出完整测试代码。
```

## 预期输出

- 测试场景清单（happy path + 边界 + 错误路径）
- 可运行的测试代码，每个用例有描述性名称
- 标注哪些场景需要人工补充（如：依赖真实外部服务的集成测试）

## 备注

- 版本：v1.0
- 已知限制：对强依赖全局状态的代码难以生成有效测试；建议先重构再生成测试
- 生成的测试须像对待生产代码一样审查 —— 测试通过 ≠ 逻辑正确
