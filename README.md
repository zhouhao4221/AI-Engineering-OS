# AI 驱动工程体系

定义 AI 驱动的工程团队如何思考、设计、构建和交付。

它不只是工具或文档的集合——它定义了团队如何与 AI 一起协作，让工程变得结构化、可复现、可扩展。

## 核心原则

**1. AI 是核心能力** — AI 不是可选项，它是系统的组成部分。

**2. 人类是协调者** — 人类定义约束、审查输出、做出决策。

**3. 一切皆结构化** — 非结构化的输入对 AI 毫无用处。

**4. 提示是资产** — 提示可复用、有版本控制，是系统的一部分。

## 团队模型

角色按**判断权归属**划分，而非执行任务：

| 层 | 职能 | 对应传统岗位 |
|----|------|------------|
| **判断层** | 决定做什么、结构是否正确 | 产品经理、架构师 |
| **编排层** | 把意图转化为 AI 可执行的任务 | 工程师、PM、项目经理 |
| **验证层** | 确认 AI 输出是否达到质量要求 | Tech Lead、QA、审查员 |

## 工作流程

```
需求 → 结构化输入
   ↓
AI 生成设计 / 代码 / 测试
   ↓
人类审查并调整约束
   ↓
AI 精炼输出
   ↓
标准化交付
```

## 工程规则

**规则 1** — 所有复杂逻辑必须经过 AI 辅助或由 AI 生成。

**规则 2** — 每次交付必须包含：
- AI 解释
- 风险分析
- 测试建议

**规则 3** — 提示必须可复用并受版本控制。

## AI 信息项目化

**推荐将所有 AI 相关信息（提示、上下文、规则、知识库）纳入所属项目，而非集中存放在独立系统中。**

### 为什么项目化

**变更原子性** — 代码改了，提示和规则必须同步更新。放在同一个项目里，一次提交可以覆盖所有相关改动；分开存放则需要跨系统协调，版本漂移不可避免。

**上下文完整性** — AI 的输出质量取决于它拿到的上下文。属于某个项目的上下文（领域术语、约束、决策历史）应该和项目代码在一起，这样 AI 工具可以直接读取，不需要人工搬运。

**可审查性** — `git blame` 一条提示词，能看到谁改的、为什么改、改之前是什么。独立系统的修改历史与代码历史永远是割裂的。

**团队共识** — 提示和规则代表团队对"AI 应该如何行为"的共同约定。放在主仓库，所有人都能看到、评审、提出异议——它是团队知识，不是个人资产。

**回滚一致** — 代码回滚时，提示和上下文自动回到对应版本，不需要额外操作。

### 跨项目信息共享

当多个项目需要共享相同的提示、规则或知识片段时，推荐以下两种方式：

**文件包（File Package）**

将共享内容打包为独立目录或仓库（类似 npm 包的思路），各项目通过 Git Submodule、脚本拷贝或 CI 同步引入。共享内容有自己的版本号，各项目可以选择更新时机。

```
shared-ai-context/        ← 独立仓库或目录
  prompts/common/         共用提示模板
  rules/code-review.md    跨项目通用规则
  glossary.md             统一术语表
```

**文件镜像（File Mirror）**

对于需要实时保持一致的内容，通过脚本或 CI 将源文件镜像到目标项目。镜像文件只读，修改必须回到源头。适合标准化程度高、变更频率低的规范文档。

```bash
# 示例：CI 同步共享规则到各项目
cp shared/rules/security.md project-a/ai-context/
cp shared/rules/security.md project-b/ai-context/
```

### 判断标准

| 信息类型 | 存放位置 |
|---------|---------|
| 项目特定的提示、上下文、领域知识 | 项目内 |
| 多项目共用的规范、模板、术语 | 文件包 / 文件镜像 |
| 运行时 A/B 测试、动态提示切换 | 独立提示管理系统（例外情况）|
