# UI 风格指南

本文档定义项目 UI 设计的核心规范维度。具体数值和命名由项目自行决定，但每个维度都应有明确答案，并在此记录。

AI 生成 UI 时以此文件为上下文约束；风格变更时同步更新此文件。

---

## 品牌基调

描述产品的视觉调性、目标用户和设计参考——这是所有具体决策的出发点。

- **设计语言**：专业克制 / 活泼友好 / 沉浸科技感 / 其他
- **目标用户**：B 端工具用户 / C 端消费者 / 混合
- **对标参考**：列出 2–3 个视觉风格接近的产品
- **明确排除**：不希望靠近的风格或产品

---

## 色彩

需要回答的问题：

- **主色**是什么，用于哪些场景（主操作、链接、选中态）？
- **中性色**体系是什么（背景、边框、文字三个层次）？
- **语义色**如何定义（成功 / 警告 / 错误 / 信息）？
- 颜色是否 Token 化，还是允许直接写色值？
- 是否支持深色模式？

**原则**：颜色数量越少越好。通常一套主色 + 一套中性色已经够用，语义色按需补充。

---

## 字体

需要回答的问题：

- **字体栈**是什么？系统字体还是自定义字体？
- **尺寸阶梯**有几档，分别对应哪些场景（正文 / 辅助 / 标题）？
- **字重**如何分配（正文用几，标题用几）？
- 字体相关属性是否 Token 化？

**原则**：尺寸档位不超过 5–6 档，字重变化控制在 2–3 档，避免视觉噪音。

---

## 间距

需要回答的问题：

- **基准单位**是多少（通常 4px 或 8px）？
- 间距值是否只取基准单位的整数倍？
- 组件内边距、组件间距、模块间距各自的典型值是多少？

**原则**：间距规律性比具体数值更重要。全局用同一套节奏，视觉才能整齐。

---

## 圆角

需要回答的问题：

- 整体风格偏向克制（小圆角）还是圆润（大圆角）？
- 不同类型组件（按钮 / 输入框 / 卡片 / 弹窗）各用多大圆角？
- 是否有全圆场景（头像、徽标）？

**原则**：圆角传达性格。专业工具偏小，消费类产品偏大。同一产品内圆角梯度不超过 3 档。

---

## 阴影与层级

需要回答的问题：

- 主要用阴影还是边框来表达层级？
- 哪些场景需要阴影（浮层、模态框、卡片）？
- 阴影的颜色、模糊度、偏移量大概是什么范围？

**原则**：能用边框解决的层级关系就不用阴影，阴影越少界面越干净。

---

## 图标

需要回答的问题：

- 使用哪个图标库？（统一一套，禁止混用）
- 默认尺寸是多少，在不同密度场景下如何变化？
- 图标颜色如何处理（继承文字色 / 独立定义）？

---

## 组件状态

所有交互组件都应明确以下状态的视觉表达：

- **默认态**：无交互时的样式
- **悬停态**：鼠标悬停，视觉给出反馈
- **激活态**：点击按下的瞬间
- **聚焦态**：键盘导航时的焦点样式（无障碍必须项）
- **禁用态**：不可交互时，样式与原因应一目了然
- **加载态**：异步操作时的过渡处理

**原则**：不能因为麻烦就跳过某个状态，尤其是聚焦态和禁用态。

---

## 动效

需要回答的问题：

- 过渡时长用多少（通常 100–200ms）？
- 缓动曲线用哪种（ease-out 适合大多数场景）？
- 哪些场景有动效，哪些明确不加动效？

**原则**：动效是反馈机制，不是装饰。克制使用，页面加载不做入场动画。

---

## 如何将此指南交给 AI

生成 UI 时，将项目填写完整的此文件作为提示前置，附加：

```
严格遵循上方风格指南。如需使用指南未涵盖的样式，先说明原因再给出代码。
```
