# 设计决策日志

记录项目中重要的架构与 UI 设计决策。目的是让团队和 AI 都能理解"为什么这样设计"，避免在未来的迭代中无意推翻已有决策。

新增决策时追加到末尾，不修改已有条目（除非决策被正式推翻，需注明推翻原因和日期）。

---

## 什么值得记录

满足以下任意一条，就应该写进来：

- **有明确的备选方案被放弃**：做了取舍，未来可能有人质疑
- **违反直觉或惯例**：选了不常见的方案，原因不写下来会被当 bug 改掉
- **影响范围广**：多个模块或团队都受这个决策约束
- **有明确的时间背景**：当时的约束条件将来可能消失，届时决策应重新评估

不需要记录：显而易见的选择、纯粹的实现细节、可以从代码直接读出意图的决定。

---

## 决策模板

```markdown
## [编号] 决策标题
**日期**：YYYY-MM
**状态**：有效 / 已推翻 / 待评审

**背景**：面临的问题是什么，为什么需要做这个决策。

**考虑的选项**：
- 选项 A：...
- 选项 B：...

**决策**：选择了哪个，理由是什么。

**影响**：这个决策会影响哪些组件或场景，有哪些已知的权衡。
```
