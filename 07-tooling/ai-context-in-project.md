# AI 信息项目化

为什么 AI 相关信息（提示、规则、上下文、知识库）应该和项目代码放在一起，而不是存在独立系统里。

---

## 问题

团队开始使用 AI 工具后，通常会出现这种情况：有效的提示存在个人聊天记录里，规则写在 Notion 或内部 Wiki，代码在 Git 仓库。三处信息各自演进，很快出现漂移——代码改了但提示没更新，规则修改了但没人知道。

**根本原因**：AI 相关信息被当成"配置"而非"代码"对待。

---

## 为什么项目化

**变更原子性** — 代码改了，提示和规则必须同步更新。放在同一个项目里，一次提交可以覆盖所有相关改动；分开存放则需要跨系统协调，版本漂移不可避免。

**上下文完整性** — AI 的输出质量取决于它拿到的上下文。属于某个项目的上下文（领域术语、约束、决策历史）应该和项目代码在一起，AI 工具可以直接读取，不需要人工搬运。

**可审查性** — `git blame` 一条提示词，能看到谁改的、为什么改、改之前是什么。独立系统的修改历史与代码历史永远是割裂的。

**团队共识** — 提示和规则代表团队对"AI 应该如何行为"的共同约定。放在主仓库，所有人都能看到、评审、提出异议——它是团队知识，不是个人资产。

**回滚一致** — 代码回滚时，提示和上下文自动回到对应版本，不需要额外操作。

---

## 项目内的组织方式

推荐在项目根目录维护以下结构：

```
project/
├── CLAUDE.md              AI 工具的项目级指导文件
├── ai-context/            （可选）较大的上下文文件单独存放
│   ├── domain-rules.md    领域规约
│   └── constraints.md     工程约束
└── 04-prompt/ 或同等目录  版本化提示资产
```

最低要求：维护一个 `CLAUDE.md`，把项目背景、约束和常用提示入口写清楚。

---

## 跨项目信息共享

当多个项目需要共享相同的提示、规则或知识片段时，推荐两种方式：

### 文件包

将共享内容打包为独立目录或仓库，各项目通过 Git Submodule、脚本拷贝或 CI 同步引入。共享内容有自己的版本号，各项目可以选择更新时机。

```
shared-ai-context/
├── prompts/common/       共用提示模板
├── rules/code-review.md  跨项目通用规则
└── glossary.md           统一术语表
```

### 文件镜像

对于需要实时保持一致的内容，通过脚本或 CI 将源文件镜像到目标项目。镜像文件只读，修改必须回到源头。适合标准化程度高、变更频率低的规范文档。

```bash
# CI 同步示例
cp shared/rules/security.md project-a/ai-context/
cp shared/rules/security.md project-b/ai-context/
```

---

## 判断标准

| 信息类型 | 存放位置 |
|---------|---------|
| 项目特定的提示、上下文、领域知识 | 项目内 |
| 多项目共用的规范、模板、术语 | 文件包 / 文件镜像 |
| 运行时 A/B 测试、动态提示切换 | 独立提示管理系统（例外情况）|

唯一需要独立系统的场景是运行时动态切换提示——这是技术需求，不是知识管理需求。
