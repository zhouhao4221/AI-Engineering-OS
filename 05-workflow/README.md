# 05 · 工作流程

标准化工程流程。在 AI 驱动团队中，工作如何从想法流转到交付。

---

## 目的

流程的一致性使 AI 输出具有可复现性。本章节记录指导工程工作开展的标准工作流程。

---

## 核心工作流程：从需求到交付

```
1. 需求
   └── 转化为结构化输入（谁/什么/约束/验收标准）

2. 设计
   └── AI 生成方案 → 人类选择并细化

3. 构建
   └── AI 生成代码/测试 → 人类审查并调整约束

4. 审查
   └── AI 审查 → 人工审查 → 完成 PR 检查清单后合并

5. 交付
   └── 标准化的发布说明、变更日志与交付后复盘
```

---

## 需求结构化模板

将模糊请求转化为 AI 可用输入：

```
目标：[一句话，期望实现什么]
背景：[相关上下文，AI 需要知道什么]
约束：[不能做什么 / 必须兼容什么]
验收标准：[如何判断完成]
排除范围：[明确不在本次任务内的内容]
```

不能填完以上所有字段，说明需求本身还需要澄清。

---

## Sprint 工作流中的 AI 检查点

| 环节 | AI 参与方式 | 人类保留什么 |
|------|-----------|------------|
| 规划 | 生成任务拆解初稿、估时参考 | 优先级决策、范围裁定 |
| 站会 | 生成进度摘要、识别阻塞模式 | 协调与决策 |
| 审查 | 初筛代码、生成审查评论 | 最终合并裁决 |
| 复盘 | 汇总周期数据、提炼改进项 | 判断哪些值得跟进 |

---

## 事故响应流程

1. **收集**：整理错误日志、堆栈、最近变更记录
2. **AI 分析**：提交结构化上下文，让 AI 生成根因假设列表
3. **人工验证**：逐一排查假设，AI 辅助查找相关代码
4. **修复**：AI 生成修复方案，人工审查后部署
5. **复盘**：记录根因和修复过程，更新相关提示或规范

**规则**：事故根因定性必须由人工确认，AI 假设不直接作为定论。

---

## 发布工作流

```
代码冻结 → AI 生成变更日志初稿 → 人工审查 → 发布说明定稿 → 部署 → 交付后监控
```

AI 生成：变更日志、发布说明、受影响模块清单  
人工确认：发布时机、回滚预案、对外沟通内容

---

## 关键规则

非结构化输入只能在第 1 步进入工作流程。到第 2 步时，一切都必须已结构化。
